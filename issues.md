# Issues — แอพนัดหมายรับบริการ

แตกจาก `prd.html` เป็น tracer-bullet vertical slices — แต่ละ slice ตัดทะลุทุกชั้น (data → API → UI → test) และ demo/verify ได้ด้วยตัวเอง แบ่งเป็น 4 เฟส

> ไม่มี issue tracker ตั้งค่าในโปรเจกต์นี้ จึงส่งมอบเป็นไฟล์ `issues.md` + `issues.html` แทนการ publish ขึ้น tracker

**สถานะโค้ดปัจจุบัน:** `server.ts` (API ครบ), `data/*.json`, `public/styles.css` มีแล้ว — ขาดหน้าเว็บฝั่ง client (`index.html`, `admin.html`, JS) และชุดเทส แต่ละ slice ระบุว่าต้องเติมอะไร

## เฟส 1 — แกนการจองของลูกค้า

### Slice 1 — โครงเดินได้ + แสดงรายการบริการ
**Type:** AFK · **Blocked by:** None — can start immediately
**User stories:** ลูกค้าเห็นรายการบริการพร้อมราคา/ระยะเวลา

#### What to build
เซิร์ฟเวอร์เสิร์ฟหน้าลูกค้า (public) ที่โหลดแล้วดึง `GET /api/services` มาแสดงรายการ**บริการ**ทั้งหมดพร้อมราคาและระยะเวลา เป็นเส้นทางบางๆ ที่ครบ data → API → UI เพื่อยืนยันว่าทั้งสามชั้นต่อกันติด

#### Acceptance criteria
- [ ] เปิด `http://localhost:3000` เห็นหน้าลูกค้าภาษาไทย
- [ ] `GET /api/services` คืนบริการจาก `data/services.json`
- [ ] หน้าเว็บแสดงบริการทุกตัวพร้อมชื่อ/ราคา/ระยะเวลา
- [ ] รันด้วย `node server.ts` ได้โดยไม่ต้องลง dependency

#### Blocked by
None - can start immediately

### Slice 2 — แสดงช่องเวลาว่าง
**Type:** AFK · **Blocked by:** Slice 1
**User stories:** เลือกวัน, เห็นเฉพาะช่องเวลาว่างที่จองได้จริง, เวลาจบคำนวณอัตโนมัติ

#### What to build
ลูกค้าเลือก**บริการ**และวัน → หน้าเว็บเรียก `GET /api/availability?date&serviceId` แล้วแสดง**ช่องเวลาว่าง**เป็นปุ่มกริด ตรรกะคำนวณ (โมดูล time-rules) ไล่จากเวลาเปิดทีละช่อง ตัดช่องที่ทำให้เวลาจบเกินเวลาปิด ตัดช่องที่เป็นอดีต (ถ้าเป็นวันนี้) และตัดช่องที่ชนนัด**ยืนยันแล้ว**

#### Acceptance criteria
- [ ] เลือกบริการ + วัน แล้วเห็นช่องเวลาว่างเป็นกริด
- [ ] ช่องที่ทำให้เวลาจบเกินเวลาปิด (เช่น 18:00) ไม่ปรากฏ
- [ ] ถ้าเลือกวันนี้ ช่องเวลาที่ผ่านไปแล้วไม่ปรากฏ
- [ ] ไม่มีช่องว่างเลย → แสดงข้อความ "ไม่มีเวลาว่างในวันนี้"

#### Blocked by
- Slice 1

### Slice 3 — จองนัด + กฎคิวเดียว
**Type:** AFK · **Blocked by:** Slice 2
**User stories:** กรอกชื่อ+เบอร์, ใส่หมายเหตุ, เห็นข้อความยืนยันเมื่อสำเร็จ, ถูกปฏิเสธชัดเจนเมื่อเวลาถูกจองไปก่อน

#### What to build
ลูกค้าเลือกช่องเวลา กรอกชื่อ+เบอร์+หมายเหตุ แล้วยืนยัน → `POST /api/appointments` สร้าง**นัดหมาย**สถานะ "ยืนยันแล้ว" โดยเวลาจบ = เวลาเริ่ม + ระยะเวลาบริการ บังคับ**กฎคิวเดียวทั้งร้าน**: ปฏิเสธ (409) ถ้าเวลาทับกับนัดที่ยืนยันแล้ว

กฎทับซ้อน (จาก prototype): `aStart < bEnd && aEnd > bStart` เทียบเฉพาะนัด `status === "confirmed"` — นัดติดกันพอดี (back-to-back) ไม่ถือว่าชน เวลาเก็บเป็น ISO ท้องถิ่นความกว้างคงที่ `YYYY-MM-DDTHH:mm` เทียบ string ได้ตรงเวลาจริง

#### Acceptance criteria
- [ ] จองสำเร็จบันทึกลง `appointments.json` สถานะ "ยืนยันแล้ว" + แสดงสรุปบริการ/วัน/เวลา
- [ ] เวลาจบของนัด = เวลาเริ่ม + ระยะเวลาบริการ
- [ ] จองเวลาที่ทับนัดยืนยันแล้ว → ตอบ 409 + ข้อความไทยชัดเจน
- [ ] นัดติดกันพอดี (back-to-back) จองได้ ไม่ถือว่าชน
- [ ] หลังจองสำเร็จ ช่องเวลานั้นหายจากช่องเวลาว่าง

#### Blocked by
- Slice 2

## เฟส 2 — จัดการร้าน (Admin)

### Slice 4 — เข้าระบบร้าน + ดูนัดรายวัน
**Type:** AFK · **Blocked by:** Slice 1
**User stories:** เข้าหน้าจัดการด้วยรหัสผ่าน, ดูนัดของวันที่เลือกเรียงตามเวลา, เห็นชื่อ/เบอร์/บริการ/หมายเหตุ

#### What to build
หน้า `/admin` มี gate รหัสผ่าน เมื่อผ่านแล้วเลือกวันแล้วเรียก `GET /api/admin/appointments?date` (ส่ง header `x-admin-pass`) แสดงรายการ**นัดหมาย**ของวันนั้นเรียงตามเวลา พร้อมชื่อบริการ ข้อมูลลูกค้า และ**สถานะ** รหัสผ่านมาจาก env `ADMIN_PASSCODE` หรือ `config.json`

#### Acceptance criteria
- [ ] `/admin` ขอรหัสผ่าน — ใส่ผิดตอบ 401 และไม่แสดงข้อมูล
- [ ] รหัสถูก เห็นรายการนัดของวันที่เลือก เรียงตามเวลา
- [ ] แต่ละรายการแสดง เวลา–เวลา / บริการ / ชื่อ / เบอร์ / หมายเหตุ / สถานะ
- [ ] วันที่ไม่มีนัด → แสดง "ยังไม่มีนัดในวันนี้"

#### Blocked by
- Slice 1

### Slice 5 — ยกเลิกนัด (soft-delete)
**Type:** AFK · **Blocked by:** Slice 3, Slice 4
**User stories:** ยกเลิกนัดได้, นัดที่ยกเลิกเก็บเป็นประวัติไม่ลบทิ้ง, เวลาที่ยกเลิกกลับมาว่างให้จองใหม่

#### What to build
ปุ่ม "ยกเลิกนัด" ที่แต่ละรายการในหน้าร้าน → `POST /api/admin/cancel` เปลี่ยน**สถานะ**เป็น "ยกเลิก" แบบ soft-delete (ไม่ลบ record) รายการยังแสดงด้วย badge แดง และเพราะกฎทับซ้อนนับเฉพาะนัดยืนยันแล้ว เวลานั้นจึงกลับมาเป็น**ช่องเวลาว่าง**ทันที

#### Acceptance criteria
- [ ] กดยกเลิก → สถานะเปลี่ยนเป็น "ยกเลิก" โดยไม่ลบรายการ
- [ ] นัดที่ยกเลิกแสดงด้วย badge แดง ยังเห็นในประวัติ
- [ ] เวลาของนัดที่ยกเลิกกลับมาจองได้ (โผล่ในช่องเวลาว่างอีกครั้ง)
- [ ] ต้องมีรหัสผ่าน — ไม่มี → ตอบ 401

#### Blocked by
- Slice 3
- Slice 4

## เฟส 3 — ตั้งค่า & ความถูกต้องภาษาไทย

### Slice 6 — ตั้งค่าบริการ/เวลาทำการผ่าน config
**Type:** AFK · **Blocked by:** Slice 2
**User stories:** ตั้งเวลาทำการและรายการบริการได้ผ่านไฟล์ config

#### What to build
ทำให้รายการ**บริการ** ชื่อร้าน และ**เวลาทำการ** (เวลาเปิด/ปิด/ช่วงละกี่นาที) ขับเคลื่อนจาก `services.json` + `config.json` ทั้งหมด การแก้ไฟล์ต้องสะท้อนทั้งหน้าเว็บและการคำนวณช่องเวลาว่างโดยไม่ต้องแก้โค้ด

#### Acceptance criteria
- [ ] เพิ่ม/ลบบริการใน `services.json` → สะท้อนหน้าลูกค้าโดยไม่แตะโค้ด
- [ ] แก้เวลาเปิด/ปิด/ช่วงเวลาใน `config.json` → ช่องเวลาว่างเปลี่ยนตาม
- [ ] ชื่อร้านจาก config แสดงบนหน้าเว็บ
- [ ] รหัสผ่านร้านอ่านจาก env `ADMIN_PASSCODE` หรือ `config.json`

#### Blocked by
- Slice 2

### Slice 7 — แสดงผลไทย พ.ศ. + validation
**Type:** AFK · **Blocked by:** Slice 3
**User stories:** เวลาจบคำนวณอัตโนมัติ, ไม่จองเวลาที่ผ่านไปแล้ว, กรอกเบอร์เพื่อให้ติดต่อกลับได้

#### What to build
ขัดเกลาความถูกต้อง: หน้าเว็บแสดงวันที่/เวลาแบบไทย พ.ศ. (`toLocaleDateString('th-TH')`) แต่เก็บข้อมูลเป็น ISO; ตรวจสอบ input ทั้งฝั่ง client และ server — เบอร์โทร, ชื่อว่าง, และกันการจองเวลาที่ผ่านไปแล้ว

#### Acceptance criteria
- [ ] หน้าเว็บแสดงวันที่แบบไทย พ.ศ.
- [ ] กรอกเบอร์ผิดรูปแบบ → error ทั้ง client และ server (400)
- [ ] ชื่อว่าง → ปฏิเสธการจอง
- [ ] จองเวลาที่ผ่านไปแล้ว → ปฏิเสธ (400) แม้ยิงตรงไปที่ API

#### Blocked by
- Slice 3

## เฟส 4 — คุณภาพ & การทดสอบ

### Slice 8 — แยกโมดูล time-rules + ชุดเทส
**Type:** AFK · **Blocked by:** Slice 3
**User stories:** (รองรับทุก story เรื่องช่องเวลาว่าง/กฎคิวเดียวให้เชื่อถือได้)

#### What to build
แยกตรรกะเวลาเป็นโมดูลบริสุทธิ์ **time-rules** (`timeToMin`, `minToTime`, `conflicts`, `computeAvailability`) ที่ไม่มี file IO เพื่อทดสอบแยกได้ แล้วเขียนชุดเทสด้วย `node:test` + `node:assert` (มากับ Node ไม่ต้องลง dependency) — เป็นชุดเทสแรกของโปรเจกต์

#### Acceptance criteria
- [ ] ตรรกะเวลาอยู่ในโมดูลเดียว ไม่มี file IO
- [ ] เทส `conflicts`: นัดทับ → true, นัดติดกันพอดี → false, นัดที่ยกเลิก → ไม่นับ
- [ ] เทส `computeAvailability`: ตัดช่องชน / นอกเวลาทำการ / อดีต / บริการยาวตัดช่องท้าย
- [ ] `node --test` ผ่านโดยไม่ต้องลง dependency

#### Blocked by
- Slice 3

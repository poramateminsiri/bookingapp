1. Problem Statement — ปัญหา

ร้านบริการขนาดเล็ก (เช่น ร้านตัดผม ร้านนวด คลินิก) รับนัดลูกค้าผ่านโทรศัพท์หรือแชท ทำให้:

  • 🔁 จองชนกัน — รับสองนัดในเวลาเดียวกันโดยไม่รู้ตัว เพราะไม่มีที่บันทึกกลาง
  • ⏰ ลูกค้าไม่รู้เวลาว่าง — ต้องถาม-ตอบไปมาว่าช่วงไหนยังจองได้
  • 📒 ไม่มีบันทึกที่ค้นได้ — ดูรายการนัดของวันนี้/ประวัติย้อนหลังลำบาก

เจ้าของร้านอยากได้ที่เดียวที่ลูกค้ากดจองเองได้ และร้านเห็นนัดทั้งหมดในที่เดียว โดยระบบกันเวลาชนให้อัตโนมัติ

2. Solution — แนวทางแก้

เว็บแอปภาษาไทย 2 หน้า:

  • 🧑‍💻 หน้าลูกค้า (public) — เลือกบริการ เลือกวัน ระบบแสดงเฉพาะช่องเวลาว่าง กรอกชื่อ+เบอร์ แล้วยืนยันนัดหมาย
  • 🛠️ หน้าจัดการร้าน (/admin) — ใส่รหัสผ่าน ดูรายการนัดของแต่ละวัน และยกเลิกนัดได้

กฎเดียวที่บังคับ: คิวเดียวทั้งร้าน — ทั้งร้านรับได้ทีละ 1 นัดต่อช่วงเวลา ระบบปฏิเสธการจองที่เวลาทับกับนัดที่ยืนยันแล้ว เวลาจบของนัดคำนวณอัตโนมัติจากระยะเวลาของบริการ

💡 ตรรกะหลัก (กฎทับซ้อน, เก็บ JSON, สถานะ) reuse จาก booking-template เดิม แต่เปลี่ยนหน้าตาจาก CLI → เว็บ

4. User Stories

ติ๊กเพื่อทำเครื่องหมายเสร็จ — สถานะจะถูกบันทึกในเบราว์เซอร์

ฝั่งลูกค้า

  • ในฐานะลูกค้า ฉันอยากเห็นรายการบริการพร้อมราคาและระยะเวลา เพื่อเลือกบริการที่ต้องการได้
  • ในฐานะลูกค้า ฉันอยากเลือกวันที่ต้องการนัด เพื่อดูเวลาว่างของวันนั้น
  • ในฐานะลูกค้า ฉันอยากเห็นเฉพาะช่องเวลาว่างที่จองได้จริง เพื่อไม่ต้องเดาว่าเวลาไหนเต็ม
  • ในฐานะลูกค้า ฉันอยากให้เวลาจบคำนวณให้อัตโนมัติตามบริการ เพื่อไม่ต้องกรอกเวลาเลิกเอง
  • ในฐานะลูกค้า ฉันอยากกรอกชื่อและเบอร์โทร เพื่อให้ร้านติดต่อกลับได้
  • ในฐานะลูกค้า ฉันอยากใส่หมายเหตุเพิ่มเติม เพื่อแจ้งความต้องการพิเศษ
  • ในฐานะลูกค้า ฉันอยากเห็นข้อความยืนยันเมื่อจองสำเร็จ เพื่อมั่นใจว่านัดถูกบันทึก
  • ในฐานะลูกค้า ฉันอยากถูกปฏิเสธอย่างชัดเจนถ้าเวลาที่เลือกถูกจองไปก่อน เพื่อเลือกเวลาใหม่ได้ทันที
  • ในฐานะลูกค้า ฉันไม่อยากจองเวลาที่ผ่านไปแล้ว เพื่อกันความผิดพลาด

ฝั่งร้าน (Admin)

  • ในฐานะเจ้าของร้าน ฉันอยากเข้าหน้าจัดการด้วยรหัสผ่าน เพื่อกันคนนอกเห็นข้อมูลลูกค้า
  • ในฐานะเจ้าของร้าน ฉันอยากดูรายการนัดของวันที่เลือก เรียงตามเวลา เพื่อเตรียมงานล่วงหน้า
  • ในฐานะเจ้าของร้าน ฉันอยากเห็นชื่อ เบอร์ บริการ และหมายเหตุของแต่ละนัด เพื่อให้บริการได้ถูกต้อง
  • ในฐานะเจ้าของร้าน ฉันอยากยกเลิกนัดได้ เพื่อจัดการกรณีลูกค้าขอเลื่อน/ยกเลิก
  • ในฐานะเจ้าของร้าน ฉันอยากให้นัดที่ยกเลิกถูกเก็บเป็นประวัติ (ไม่ลบทิ้ง) เพื่อตรวจสอบย้อนหลังได้
  • ในฐานะเจ้าของร้าน ฉันอยากให้เวลาที่ยกเลิกแล้วกลับมาว่างให้จองใหม่ได้ เพื่อไม่เสียคิว
  • ในฐานะเจ้าของร้าน ฉันอยากตั้งเวลาทำการและรายการบริการได้ผ่านไฟล์ config เพื่อปรับให้ตรงกับร้าน
5. Implementation Decisions

สถาปัตยกรรม

  • ADR-0001 เว็บแอปแทน CLI — reuse ตรรกะหลักของเทมเพลตเป็น backend
  • ADR-0002 Node.js built-in ล้วน (node:http, node:fs) รัน .ts ตรงด้วย Node 24 type-stripping ไม่มี framework/build step
  • เก็บข้อมูลเป็นไฟล์ JSON: appointments.json, services.json, config.json

โมดูล (deep modules)

โมดูลหน้าที่ / อินเทอร์เฟซ
time-rules
ฟังก์ชันบริสุทธิ์		timeToMin, minToTime, conflicts(start,end,appts), computeAvailability(date,service,appts,config) — ไม่มี IO ทดสอบแยกได้
appointment-servicecreateAppointment(body) (validate + กฎคิวเดียว + persist), cancel(id)
storagereadJSON / writeAppointments — ห่อ file IO
http-serverrouting บางๆ + serve static (ชั้นบาง ไม่มี business logic)

ฟิลด์ชนิด / ตัวอย่าง

API contract

Endpointหน้าที่
GET /api/servicesรายการบริการ
GET /api/configชื่อร้าน + เวลาทำการ
GET /api/availability?date&serviceIdคืน { slots: ["09:00", …] }
POST /api/appointmentsสร้างนัด → 201 / 409 (ชนคิว) / 400
GET /api/admin/appointments?dateรายการนัด (ต้องมี header x-admin-pass)

กฎสำคัญ

ทับซ้อนเมื่อ aStart < bEnd && aEnd > bStart เทียบเฉพาะนัด ยืนยันแล้ว · เวลาเก็บเป็น ISO ท้องถิ่นความกว้างคงที่ จึงเทียบแบบ string ได้ตรงตามเวลาจริง · นัดติดกันพอดี (back-to-back) ไม่ถือว่าชน
6. Testing Decisions

เทสที่ดี = ทดสอบพฤติกรรมภายนอก ไม่ใช่รายละเอียดภายใน — ป้อน input แล้วตรวจ output โดยไม่ผูกกับวิธี implement

โมดูลที่จะเทส

  • หลัก time-rules — ฟังก์ชันบริสุทธิ์ เทสง่ายและคุ้มที่สุด:
    • conflicts: นัดทับกัน → true, นัดติดกันพอดี → false, นัดที่ยกเลิกแล้ว → ไม่นับ
    • computeAvailability: ตัดช่องที่ชน, ตัดช่องนอกเวลาทำการ, ตัดเวลาที่ผ่านมาแล้ว, บริการยาวกว่าตัดช่องท้ายที่เลยเวลาปิด
  • รอง appointment-service — เทสผ่าน storage จำลอง: สร้างสำเร็จ, ปฏิเสธเมื่อชน (409), ปฏิเสธ input ไม่ถูกต้อง (400)

Prior art

เทมเพลตต้นทางไม่มีชุดเทส — โมดูล time-rules เป็นชุดเทสแรกของโปรเจกต์ ใช้ node:test + node:assert (มากับ Node ไม่ต้องลง dependency)

7. Out of Scope
  • ❌ บัญชีลูกค้า / ระบบ login ฝั่งลูกค้า (ระบุแค่ชื่อ+เบอร์)
  • ❌ พนักงาน/ผู้ให้บริการหลายคน (ร้านคิวเดียว — ดู ADR/CONTEXT)
  • ❌ การแจ้งเตือน SMS/อีเมล/LINE
  • ❌ การชำระเงิน/มัดจำออนไลน์
  • ❌ หลายสาขา / multi-tenant
  • ❌ การเลื่อนนัด (reschedule) — ทำได้โดยยกเลิกแล้วจองใหม่
  • ❌ ฐานข้อมูลจริง (ใช้ไฟล์ JSON)
